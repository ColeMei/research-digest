### 1. [VIPER-MCP: Detecting and Exploiting Taint-Style Vulnerabilities in Model Context Protocol Servers](https://arxiv.org/abs/2605.21392)
**Authors:** Pengyu Sun, Qishu Jin, Enhao Huang, Zifeng Kang, Xin Liu, Dakun Shen, Song Li · arXiv cs.CR · submitted 2026-05-20

VIPER-MCP is the first end-to-end automated auditing framework for MCP servers that not only detects vulnerabilities but dynamically confirms their exploitability by generating concrete proof-of-concept prompts. The core insight is that MCP tool handlers create a direct path from natural-language input to privileged sinks — shell execution, filesystem access, network calls — making taint-style bugs especially dangerous. Detection uses a two-pass static analysis strategy: a standard taint pass followed by an anchor-query pass that augments each alert with function-level structural context, resolving file-level artifacts to specific tool handlers and producing vulnerability-anchored call chains. Exploitability confirmation uses a feedback-driven prompt evolution mechanism with a dual-mutator scheduler that independently corrects tool-selection drift and deepens parameter penetration. Applied to 39,884 open-source MCP server repositories on GitHub, VIPER-MCP discovered **106 zero-day vulnerabilities**.

**Methodology:** Two-pass static taint analysis (standard + anchor-query) → feedback-driven prompt evolution with dual-mutator scheduling for dynamic PoC confirmation; evaluated against 39,884 real-world MCP server repos.

**Why it matters:** MCP is fast becoming the standard integration layer for agentic AI systems; this paper directly characterizes and demonstrates end-to-end exploitability of that emerging attack surface at scale.

### 2. [FuzzingBrain V2: A Multi-Agent LLM System for Automated Vulnerability Discovery and Reproduction](https://arxiv.org/abs/2605.21779)
**Authors:** Ze Sheng, Zhicheng Chen, Qingxiao Xu, Kewen Zhu, Jeff Huang · Texas A&M University · arXiv cs.CR/cs.SE · submitted 2026-05-20

FuzzingBrain V2 tackles the three central weaknesses of existing LLM-based vulnerability finders: high false-positive rates, suboptimal granularity (function-level misses context; line-level lacks it), and inability to reason across complex cross-function control flow. The system introduces a novel *Suspicious Point* abstraction — a control-flow-based unit between function and line granularity — as the target for vulnerability localization, and layers a multi-agent reasoning pipeline on top of Google's OSS-Fuzz, ensuring every reported finding is fuzzer-reproducible before it leaves the system. A dedicated reproduction agent synthesizes a triggering PoC and confirms it on OSS-Fuzz before the report is surfaced. Deployed since early 2026, the system has discovered **41 previously unknown vulnerabilities** across 19 projects; 26 are confirmed, 23 patched, and two have been assigned CVEs (CVE-2026-23874 and CVE-2026-23952 in ImageMagick).

**Methodology:** Multi-agent LLM pipeline (localizer → reasoning agent → reproduction agent) using a control-flow-based Suspicious Point abstraction; all findings validated via OSS-Fuzz before reporting; continuous deployment mode.

**Why it matters:** Grounding every finding in a fuzzer-reproducible crash eliminates the false-positive problem that limits practical deployment of LLM-based detectors — the reproducibility-first architecture is directly adoptable.

### 3. [Project Glasswing: An Initial Update](https://www.anthropic.com/research/glasswing-initial-update)
**Authors:** Anthropic Frontier Red Team · Anthropic Research Blog · 2026-05-22

Anthropic published the first substantive data from Project Glasswing, which deploys Claude Mythos Preview autonomously to approximately 50 trusted partner organizations to audit their production codebases. After one month, partners have collectively identified **more than 10,000 high- or critical-severity vulnerabilities**, most of them previously unknown. Cloudflare found 2,000 bugs across critical-path systems (400 rated high/critical), with a false-positive rate their team considers better than human testers. Mozilla identified and fixed 271 vulnerabilities in Firefox 150 during the testing period. The update flags a new operational bottleneck: the average time to patch a high/critical finding from Mythos Preview is two weeks, meaning discovery throughput has far outpaced remediation capacity. Anthropic's Frontier Red Team blog provides technical write-ups for a subset of patched findings, including a fully autonomous 17-year-old FreeBSD NFS root-execution vulnerability — no human input after the initial tasking.

**Methodology:** Autonomous deployment of Claude Mythos Preview to partner codebases; findings triaged and verified by partner security teams; technical write-ups published for a subset of patched bugs.

**Why it matters:** This is the first large-scale published measurement of AI-driven vulnerability discovery in real production systems, with confirmed findings and patch rates — a direct benchmark for the practical AI4Sec research agenda.

### 4. [Blind Spots in the Guard: How Domain-Camouflaged Injection Attacks Evade Detection in Multi-Agent LLM Systems](https://arxiv.org/abs/2605.22001)
**Authors:** Aaditya Pai · Data Science Institute, Columbia University · arXiv cs.CR/cs.CL · submitted 2026-05-21

Current injection detectors are calibrated on static, template-based payloads that announce themselves as override directives. This paper shows that payloads crafted to mimic the vocabulary and authority structures of the target document — "domain-camouflaged injection" — completely defeat standard detectors: detection rates drop from **93.8% to 9.7%** on Llama 3.1 8B and from 100% to 55.6% on Gemini 2.0 Flash. The paper formalizes this as the *Camouflage Detection Gap* (CDG) and demonstrates it is large and statistically significant across 45 tasks spanning three domains (legal, medical, financial) and two model families. The attack requires no access to the detector's internals — only knowledge of the domain vocabulary — making it a practical threat to any agentic pipeline that relies on a text-classification guard.

**Methodology:** Domain-camouflaged payloads constructed by mimicking target document vocabulary and authority patterns; evaluated against standard prompt injection classifiers over 45 tasks × 3 domains × 2 model families; CDG quantified and tested for statistical significance.

**Why it matters:** Defense-in-depth for multi-agent pipelines depends on reliable injection detection; this paper identifies a systematic blind spot and is a direct call to action for any research building agentic security tooling.

### 5. [ContraFix: Agentic Vulnerability Repair via Differential Runtime Evidence and Skill Reuse](https://arxiv.org/abs/2605.17450) *(~May 17)*
**Authors:** Simiao Liu, Fang Liu, Li Zhang, Yang Liu · Beihang University; Yinghao Zhu · The University of Hong Kong · arXiv cs.SE/cs.CR · submitted ~2026-05-17

Existing LLM agents for automated vulnerability repair (AVR) fail mainly through *semantic misunderstanding*: they fix symptoms rather than root causes because a single crash trace tells you where the program failed, not which state transition separates crashing behavior from safe execution. ContraFix addresses this by instrumenting two executions — the crashing run and a safe near-miss — and extracting the differential between them as structured evidence to steer the repair agent toward the correct causal variable or code path. A *skill reuse library* then retains the diagnostic evidence from each successful repair so that structurally similar future cases don't require re-diagnosis. The paper reports substantial improvements over prior AVR agents on real-world CVE benchmarks, with patches that pass both functional regression tests and semantic correctness checks.

**Methodology:** Differential runtime instrumentation comparing crashing and safe execution traces; LLM repair agent conditioned on differential evidence; skill reuse library accumulates repair patterns cross-vulnerability.

**Why it matters:** Automated patching is only useful if it fixes root causes — this paper directly closes the most common failure mode of current AVR systems and the reuse mechanism is applicable to any agentic repair pipeline.

### [GRIEF: Continuous Discovery of Vulnerabilities in LLM Serving Systems with Fuzzing](https://arxiv.org/abs/2605.11202) *(near-window, ~May 11; not in prior digest)*
**Authors:** Not individually disclosed in search-accessible metadata · arXiv cs.CR · ~2026-05-11

GRIEF is a greybox fuzzer for LLM inference engines that treats timed multi-request traces as first-class fuzzing inputs, reflecting the shared-state behaviors — KV-cache isolation, batching, prefix sharing, speculative decoding — that only manifest under realistic concurrent workloads. Lightweight oracles detect crashes, hangs, performance pathologies, and silent output corruption (cross-request KV-cache leakage, token drops). Applied to vLLM and SGLang, GRIEF discovered **15 vulnerabilities**, with 10 confirmed by engine developers including **2 CVEs** for KV-cache isolation failures and cross-request performance issues.

**Methodology:** Greybox fuzzing with timed multi-request traces as the fuzzing primitive; oracles for four failure classes; log-probability checks for silent output corruption; evaluated on vLLM and SGLang.

**Why it matters:** LLM inference infrastructure is becoming security-critical; GRIEF demonstrates it has an exploitable attack surface that standard model/API tests miss entirely.

### [We Hardened zizmor's GitHub Actions Static Analyzer](https://blog.trailofbits.com/2026/05/22/we-hardened-zizmors-github-actions-static-analyzer/)
**Authors:** Trail of Bits · Trail of Bits Blog · 2026-05-22

When GitHub Actions added YAML anchor support in September 2025, a high-value slice of the CI/CD ecosystem began writing workflows that zizmor — the leading GitHub Actions security static analyzer — could only analyze on a best-effort basis. Trail of Bits collaborated with the zizmor maintainers over three months to bring anchor coverage to full spec compliance: fixing parsing bugs that caused crashes and wrong-location findings, resolving deserialization edge cases on valid-but-unusual workflows, and aligning the expression evaluator with GitHub's Known Answer Tests. The work produced 20 filed issues and 15 merged PRs and was validated against **41,253 real-world GitHub Actions workflows** from high-value open-source projects.

**Methodology:** Systematic fuzzing and conformance testing of zizmor's YAML anchor parser against GitHub Actions' own known-answer tests; large-scale validation on 41,253 real-world workflows.

**Why it matters:** zizmor is a key tool for automated CI/CD security analysis; this hardening work directly improves the reliability of automated pipeline security auditing at scale.

### [The Range Shrinks, the Threat Remains: Re-evaluating LLM Package Hallucinations on the 2026 Frontier-Model Cohort](https://arxiv.org/abs/2605.17062) *(~May 16)*
**Authors:** Aleksandr Churilov · Independent Researcher · arXiv cs.CR · submitted ~2026-05-16

This paper evaluates whether newer frontier models have reduced the slopsquatting attack surface: across 199,845 paired Python and JavaScript prompts validated against PyPI and npm master lists, hallucination rates range from **4.62%** (Claude Haiku 4.5) to **6.10%** (GPT-5.4-mini) across five models released October 2025–March 2026. The main finding is that absolute rates have declined modestly since prior benchmarks but remain high enough to sustain a practical slopsquatting attack surface, especially in JavaScript. The paper also identifies that hallucinated names cluster around plausible-sounding utility packages — the same distribution that attackers register — rather than random strings.

**Methodology:** 199,845 Python + JavaScript code-generation prompts across five frontier models; hallucinated package names validated against live PyPI and npm registries; statistical comparison against prior-cohort benchmarks.

**Why it matters:** AI-generated code in security tooling is increasingly common; quantifying the hallucination-driven supply-chain attack surface is directly relevant to secure-coding research.

### [Running Codex Safely at OpenAI](https://openai.com/index/running-codex-safely/)
**Authors:** OpenAI Security Engineering · OpenAI Blog · 2026-05-08

OpenAI published an engineering post describing the sandboxing architecture for Codex, their cloud-based autonomous coding agent. The post details how every Codex task runs in a per-task microVM with no network egress by default, filesystem mount namespaces scoped to the task workspace, and a seccomp-bpf syscall filter. The post also covers the process for safe "network-enabled" mode in tasks that require package installation, including a DNS allowlist and proxy layer. While primarily operational, the post surfaces concrete threat models for autonomous coding agents and explains the tradeoffs between security isolation and usability.

**Methodology:** Architectural description of per-task microVM isolation with seccomp-bpf syscall filtering, network egress controls, and audit logging for Codex.

**Why it matters:** As autonomous code agents become prevalent in security tooling, their own isolation boundaries are an emerging attack surface; this post is a concrete reference design for agent sandboxing.

**Authors:** OpenAI Security · OpenAI Blog · 2026-05-07

OpenAI announced GPT-5.5-Cyber, a restricted-access model fine-tuned for offensive and defensive cybersecurity tasks, made available to vetted US government and critical-infrastructure organizations. The post describes a tiered access model where the base GPT-5.5 has policy constraints on detailed exploit guidance, while GPT-5.5-Cyber extends those constraints in both directions — accepting operational offensive tasking from authorized users and providing more targeted vulnerability analysis. OpenAI also published aggregate performance data showing GPT-5.5-Cyber outperforms GPT-5.5 on CTF categories and on a private evaluation set of real CVE exploitation tasks.

**Methodology:** Fine-tuning GPT-5.5 on curated cybersecurity task datasets; tiered access controls; evaluated on CTF benchmarks and a private CVE exploitation evaluation set.

**Why it matters:** Restricted-access security models are a new deployment paradigm — the access-tier architecture is relevant for thinking about responsible deployment of AI security research tools.
