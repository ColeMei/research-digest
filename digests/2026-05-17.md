## Top picks

### 1. [ExploitGym: Can AI Agents Turn Security Vulnerabilities into Real Attacks?](https://arxiv.org/abs/2605.11086)
**Authors:** Dawn Song, Nicholas Carlini, Florian Tramèr, Shangdi Yu, and 20+ co-authors from UC Berkeley, Max Planck Institute for Security and Privacy, UC Santa Barbara, Arizona State University, Anthropic, OpenAI, and Google · arXiv cs.CR · submitted 2026-05-11

ExploitGym is the first large-scale, multi-domain benchmark for evaluating whether AI agents can turn a known vulnerability trigger into a working exploit with real security impact. Its 898 instances span three attack surfaces: 520 userspace C/C++ programs (FFmpeg, OpenSSL, and others), 185 V8 JavaScript engine bugs, and 193 Linux kernel tasks requiring full privilege escalation inside a VM. Given a crash-inducing input, the agent must extend it into something with concrete impact — unauthorized file read, arbitrary code execution, or privilege escalation. Results show Claude Mythos Preview succeeds on 157 instances and GPT-5.5 on 120; even with ASLR and stack canaries enabled, frontier models retain non-trivial success rates. The gap between "find a crash" and "build a working exploit" is real but narrowing fast.

**Methodology:** 898-instance benchmark across three exploitation domains; agents receive a triggering PoC and must autonomously develop a working exploit; evaluated with and without standard binary hardening defenses.

**Why it matters:** This is the most comprehensive public measurement of AI exploitation capability to date — directly calibrates where frontier models sit on the PoC→exploit continuum, which is central to both offensive-capability evaluation and responsible-disclosure timing.

### 2. [ExploitBench: A Capability Ladder Benchmark for LLM Cybersecurity Agents](https://arxiv.org/abs/2605.14153)
**Authors:** Seunghyun Lee, David Brumley (Carnegie Mellon University / Bugcrowd) · arXiv cs.CR · submitted 2026-05-13

Existing LLM security benchmarks treat a crash as exploitation success, which collapses the hardest part of offensive work: turning a trigger into reusable control primitives and eventual code execution. ExploitBench decomposes exploitation into 16 deterministically verified flags — from coverage and crash through sandbox primitives, arbitrary read/write, control-flow hijack, and arbitrary code execution — instantiated on 41 V8 JavaScript engine bugs. Oracles use per-run challenge-response for primitives, differential execution against ground-truth binaries to measure progress, and a signal-handler proof for code execution. Results across 8 publicly deployed frontier models show that reaching vulnerable code and triggering a crash is routine, but arbitrary code execution is achieved by none of them; a private frontier model (undisclosed) achieves ACE on approximately half. The benchmark cleanly separates capability tiers that flat pass/fail metrics conflate.

**Methodology:** Fine-grained 16-flag capability ladder on 41 real V8 bugs; deterministic oracle design using challenge-response, differential execution, and signal-handler proofs; evaluated against 8 public + 1 private frontier model.

**Why it matters:** The capability-ladder framing is exactly what the field needs to avoid inflated benchmark numbers; the methodology is directly adoptable for evaluating any AI system's progress from bug trigger to full exploit.

### 3. [Google GTIG: First AI-Built Zero-Day Exploit Detected in the Wild](https://www.theregister.com/ai-ml/2026/05/11/google-says-criminals-used-ai-built-zero-day-in-planned-mass-hack-spree/5237982)
**Source:** Google Threat Intelligence Group (GTIG) · Disclosed 2026-05-11 · Covered by Bloomberg, SecurityWeek, The Register, and others

Google's Threat Intelligence Group confirmed on May 11 the first documented case of threat actors using an AI-generated zero-day exploit in a real attack: a 2FA bypass targeting a widely-deployed open-source web administration tool, caught before a planned mass-exploitation campaign and patched after coordinated vendor disclosure. GTIG identified the script as AI-generated from a cluster of fingerprints: educational inline docstrings explaining each exploit step, a hallucinated CVSS score embedded as a comment, textbook-clean Pythonic structure, a fabricated `_ANSI` color class, and an absence of the idiosyncratic variable names or comment shortcuts typical of human exploit authors. The underlying vulnerability stemmed from a hard-coded trust exception in the authentication flow — a logic bug of the kind LLMs can reliably locate from code review alone. Google explicitly stated that neither Gemini nor Anthropic's Mythos was involved; the threat actor's identity and the specific tool remain undisclosed.

**Methodology:** Threat intelligence attribution via behavioral fingerprinting of exploit code artifacts; coordinated disclosure to vendor before public announcement.

**Why it matters:** The "AI vulnerability race" moved from forecast to documented fact this week — this incident establishes that capability has already crossed the threshold where adversaries are deploying AI-generated exploits operationally.

### 4. [Defense at AI Speed: Microsoft's Multi-Model Agentic Security System Tops Leading Industry Benchmark](https://www.microsoft.com/en-us/security/blog/2026/05/12/defense-at-ai-speed-microsofts-new-multi-model-agentic-security-system-tops-leading-industry-benchmark/)
**Authors:** Taesoo Kim (VP, Agentic Security), with the Autonomous Code Security (ACS) team, MORSE, and WARP — Microsoft · Microsoft Security Blog · 2026-05-12

Microsoft disclosed MDASH (Multi-model Agentic Scanning Harness), a production-deployed vulnerability discovery system that orchestrates 100+ specialized AI agents across a five-stage pipeline: Prepare (build language-aware indices, map attack surfaces), Scan (deploy auditor agents across code paths), Validate ("debater" agents that argue for and against each finding's exploitability), Dedup (patch-based semantic deduplication), and Prove (construct and execute triggering inputs to dynamically confirm findings). The ensemble combines frontier models for reasoning, distilled models for cost efficiency, and secondary SOTA models as independent counterpoints. Quantified results: 21/21 deliberately planted vulnerabilities found with zero false positives; 96% recall on five years of confirmed MSRC cases in `clfs.sys` and 100% in `tcpip.sys`; 88.45% on the public CyberGym benchmark (currently #1 on its leaderboard). This week's Patch Tuesday included 16 CVEs MDASH found, including four Critical RCE flaws in `tcpip.sys` and `ikeext.dll`, and two multi-file kernel findings (a race-condition use-after-free across non-trivial control flow, CVE-2026-33827, and an aliasing double-free across six source files, CVE-2026-33824).

**Methodology:** Five-stage multi-agent pipeline (Prepare → Scan → Validate → Dedup → Prove) with 100+ specialized agents; evaluated on planted-bug benchmark, five-year MSRC case history, and public CyberGym leaderboard.

**Why it matters:** MDASH is the clearest evidence yet that agentic vulnerability discovery has crossed from research into production enterprise defense — and that the orchestration architecture (debater agents, dedup-before-prove) matters as much as the underlying models.

### 5. [Agentic Fuzzing: Opportunities and Challenges](https://arxiv.org/abs/2605.10074)
**Authors:** Not individually disclosed in search-accessible metadata (paper affiliation TBC) · arXiv cs.CR · submitted ~2026-05-12

This paper proposes agentic fuzzing as a new paradigm for finding logic bugs in mature codebases that resist traditional coverage-guided fuzzing — bugs that require multi-step reasoning and produce no distinctive execution signal. The approach seeds the agent with a reference bug, which the agent analyzes for root cause, hypothesizes new scenarios elsewhere in the codebase sharing that cause, and verifies each hypothesis by generating and running proof-of-concept code. The implementation, **AFuzz**, uses a four-stage agent pipeline, a scenario-coverage mechanism that deduplicates previously explored root-cause scenarios across seeds, and a DPP-MAP scheduler that orders seeds by diversity. Applied to V8 for approximately one month: 40 bugs found (3 duplicates), $35,000 in bug bounties, 2 CVEs assigned; cross-project transfer to SpiderMonkey and JavaScriptCore using V8-derived seeds yielded 19 additional bugs without restating the root-cause analysis.

**Methodology:** Historical-bug-seeded agentic loop with root-cause hypothesis generation, scenario deduplication via coverage tracking, and DPP-MAP seed scheduling; evaluated on V8, SpiderMonkey, and JavaScriptCore.

**Why it matters:** The $35k bounty and cross-engine transfer result are strong empirical signals that agentic fuzzing can operationally complement or replace manual variant analysis — a key workflow for your PoC-generation research direction.

### [Behavioral Integrity Verification for AI Agent Skills](https://arxiv.org/abs/2605.11770)
**Authors:** Yuhao Wu, Tung-Ling Li, Hongliang Liu · arXiv cs.CR · submitted 2026-05-12

Agent skills (plugins, tool-use extensions) routinely claim declared capabilities in their metadata while implementing something different in their code. This paper formalizes the Behavioral Integrity Verification (BIV) problem as a typed-set comparison between declared and actual capabilities over a shared taxonomy, then instantiates it by pairing deterministic static code analysis with LLM-assisted natural-language capability extraction. Applied to 49,943 skills from the OpenClaw registry, the deviation taxonomy finds that 80.0% of skills deviate from declared behavior — including four novel compound-threat categories (undisclosed data exfiltration, silent lateral movement, credential harvesting under benign labels, and privilege escalation via scope creep). This is a supply-chain security problem specific to agentic systems: the composition of trusted models and unverified skills creates an exploitable gap.

**Methodology:** Static code analysis + LLM capability extraction over 49,943 OpenClaw registry skills; BIV framework maps declared vs. actual capabilities over shared taxonomy.

### [A 0-Click Exploit Chain for the Pixel 10: When a Door Closes, a Window Opens](https://projectzero.google/2026/05/pixel-10-exploit.html)
**Authors:** Seth Jenkins, with Jann Horn · Google Project Zero · 2026-05-13

Seth Jenkins details a two-stage zero-click Android exploit chain for the Pixel 10, building on January's Pixel 9 research. Stage one reuses CVE-2025-54957, a vulnerability in the Dolby Unified Decoder present on virtually all Android devices because incoming Messages audio is transcribed before the user opens it. The January patch closed the Pixel 9 path through `BigWave`, but the Pixel 10's Tensor G5 chip introduced a new VPU driver at `/dev/vpu` (Chips&Media Wave677DV) whose developers made the same class of memory-management mistakes. Two hours of joint auditing with Jann Horn found a vulnerability requiring 5 lines of code to achieve arbitrary kernel read-write; the full exploit took less than a day to write. The bug was reported November 24, 2025 and patched in the February Pixel bulletin (71-day patch gap). No AI angle, but exemplary public exploit engineering.

**Methodology:** Manual source audit of the Wave677DV VPU kernel driver; two-stage exploit chain development (Dolby UDC OOBB → VPU kernel R/W); patched and disclosed via standard 90-day policy.

**Why it matters:** The Pixel 10 case demonstrates how a patch that closes one attack path can inadvertently open another through new hardware — a pattern relevant when evaluating automated patch-verification systems.

### [Go Fuzzing Was Missing Half the Toolkit. We Forked the Toolchain to Fix It.](https://blog.trailofbits.com/2026/05/12/go-fuzzing-was-missing-half-the-toolkit.-we-forked-the-toolchain-to-fix-it./)
**Authors:** Trail of Bits · Trail of Bits Blog · 2026-05-12

Trail of Bits released **GoSentry**, a fork of the Go toolchain that replaces `go test -fuzz`'s native fuzzing engine with LibAFL by default while preserving the standard harness interface so existing fuzz targets work unmodified. The native Go fuzzer lacks bug-class detectors for integer overflows, goroutine leaks, data races, and execution timeouts that are standard in C/C++ fuzzing pipelines. GoSentry injects compiler-level integer overflow checks, integrates the native Go race detector and `goleak` at fuzz time, adds Nautilus grammar-based fuzzing support, and generates a campaign coverage report in one command. Struct-aware fuzzing is also supported natively.

**Methodology:** Fork of the Go toolchain replacing the native fuzzer backend with LibAFL; compiler-inserted sanitizer passes for overflow and race detection; evaluated against existing Go fuzz harnesses without modification.

**Why it matters:** Go is increasingly common in cloud infrastructure and security tooling; GoSentry brings its fuzzing up to C-ecosystem standards, which matters when assessing attack surface on Go-based security-critical software.

### [Security Risks in Tool-Enabled AI Agents: A Systematic Analysis of Privileged Execution Environments](https://arxiv.org/abs/2605.09721)
**Authors:** Not individually disclosed in search-accessible metadata · arXiv cs.CR · IEEE COMPSAC 2026 (short paper) · ~2026-05-12

This COMPSAC 2026 short paper develops a taxonomy of security risks that arise specifically when LLM agents operate with privileged access to production tools — filesystems, shells, databases, external APIs. The authors identify three representative cloud-hosted agent deployment scenarios and map each to a structured threat model covering confidentiality, integrity, and availability dimensions. Mitigation strategies are discussed with tradeoff analysis. Primarily a definitional and taxonomic contribution; the IEEE COMPSAC venue gives it peer-reviewed status even without novel empirical results.

**Methodology:** Qualitative threat modeling across three representative privileged-agent deployment scenarios; taxonomy construction with mitigation tradeoff analysis.

### [Agentic AI and the Industrialization of Cyber Offense](https://arxiv.org/abs/2605.06713)
**Authors:** Not individually disclosed · arXiv cs.CR · submitted ~2026-05-08

This paper introduces a Three-Channel Agentic Cyber-Risk Model distinguishing between AI-assisted, AI-augmented, and AI-autonomous offensive operations, and uses the 2026 Linux kernel "Copy Fail" supply-chain incident as a case study for the third channel. It argues that commodity LLMs lower the floor for entry-level attackers while elite threat actors benefit disproportionately from AI planning and orchestration capabilities. Primarily a policy/forecast paper rather than an empirical one, but the taxonomy is clean and the case study is current.

**Methodology:** Qualitative cyber-risk model construction; case study analysis of the 2026 "Copy Fail" Linux kernel incident.
