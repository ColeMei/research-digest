## Top picks

### 1. [GitInject: Real-World Prompt Injection Attacks in AI-Powered CI/CD Pipelines](https://arxiv.org/abs/2606.09935)
**Authors:** Jafar Isbarov, Umid Suleymanov, Ilia Shumailov, Murat Kantarcioglu · arXiv cs.CR/cs.SE · submitted 2026-06-09

Modern CI/CD platforms have wired LLM agents directly into their build loops — GitHub AI Inference, Gemini CLI Actions, OpenAI Codex Actions, and Claude Code Actions all process pull-request titles, commit messages, issue bodies, and code comments before dispatching privileged tool calls. GitInject provides the first confirmed real-world demonstration that an attacker who can open a PR or post an issue can craft content that redirects the agent to exfiltrate repository secrets, modify build artifacts, or alter deployment targets entirely through natural-language injection. The paper maps the attack surface across four major platforms, documents end-to-end exploit chains on each, and shows that existing input-validation heuristics and prompt-level defenses fail uniformly because they evaluate injected content in isolation from the agent's permission context. A disclosure timeline is included for the affected vendors.

**Methodology:** End-to-end exploit construction across four CI/CD platforms; manual taint analysis from attacker-controlled input sources (PR/issue/commit) through the agent's tool-call graph; evaluation of existing mitigations against each attack chain.

**Why it matters for your work:** If you use any AI-powered CI/CD in your research pipeline — or study AI-empowered software engineering security — this paper documents the exact threat model and shows that current deployments are exploitable today.

### 2. [SecureClaw: Clawing Back Control of LLM Agents](https://arxiv.org/abs/2606.09549)
**Authors:** Yuhan Ma, Stefan Schmid · TU Berlin · arXiv cs.CR · submitted 2026-06-08

Tool-using LLM agents face two structurally distinct security failures: they may take external actions the user never approved, and they may expose sensitive plaintext (credentials, PII, API keys) inside the runtime before any output filter can intervene. SecureClaw proposes a dual-boundary architecture to address both simultaneously. At the read boundary, a trusted gateway intercepts credential and sensitive-data lookups, replaces raw values with opaque handles, and exposes only bounded, explicitly declassified summaries to the LLM's context — ensuring secrets never appear as strings the model can leak or be prompted to repeat. At the write boundary, external state-changing calls follow a PREVIEW→COMMIT protocol: the agent constructs a canonical request that a trusted executor must commit verbatim, preventing post-approval tampering or prompt-injection-induced substitution. The architecture is evaluated on a suite of authorization and confidentiality attack scenarios, demonstrating complete containment at the cost of modest latency overhead.

**Methodology:** Dual-boundary runtime architecture; opaque-handle proxy at read boundary; PREVIEW→COMMIT trusted-executor protocol at write boundary; evaluation against authorization and plaintext-leakage attack scenarios.

**Why it matters for your work:** Directly addresses the "unauthorized agentic action" problem at the architectural level — a foundational reference for anyone building or auditing security-oriented multi-agent systems.

### 3. [CyberGym-E2E: Scalable Real-World Benchmark for AI Agents' End-to-End Cybersecurity Capabilities](https://arxiv.org/abs/2606.04460)
**Authors:** Tianneng Shi, Robin Rheem, Dongwei Jiang, Mona Wang, Francisco De La Riega, Zhun Wang, Jingzhi Jiang, Alexander Cheung, Sean Tai, Jonah Cha, Jianhong Tu, Gabriel Han, Chenguang Wang, Jingxuan He, Wenbo Guo, Dawn Song · UC Berkeley and collaborators · ICML 2026 · submitted 2026-06-03 *(missed from prior digest)*

Existing AI cybersecurity evaluations assess at most one phase of the vulnerability lifecycle in isolation. CyberGym-E2E closes that gap with 920 tasks across 139 open-source projects, covering all three phases end-to-end: vulnerability detection, proof-of-concept exploit generation, and patch generation. The benchmark is constructed from OSS-Fuzz data via an automated, agent-enhanced pipeline that synthesises ground-truth PoCs, pairs of vulnerable and patched project versions, build scripts, and containerized execution environments — making it reproducible and continuously extensible. Each task is evaluated with a strict oracle: 2 points for a valid PoV (proof-of-vulnerability), 6 points for a correct patch that neutralises the PoV while passing regression tests, mirroring the AIxCC scoring rubric at open-source scale.

**Methodology:** Automated OSS-Fuzz → Docker pipeline for environment construction; full-lifecycle evaluation (detect + PoC + patch) with AIxCC-style point scoring; 920 tasks across 139 projects.

**Why it matters for your work:** This is the benchmark infrastructure for measuring AI-driven security workflows end-to-end — directly applicable to evaluating any system you build for vulnerability detection or automated patching.

### 4. [deepSURF: Detecting Memory Safety Vulnerabilities in Rust Through Fuzzing LLM-Augmented Harnesses](https://arxiv.org/abs/2506.15648)
**Authors:** Zhuobing Han, Xiaoxing Ma, et al. · Nanjing University · Accepted at IEEE Symposium on Security and Privacy 2026

Rust's ownership system eliminates most memory-safety bugs in safe code, but `unsafe` blocks remain and are poorly covered by existing fuzzers because writing effective harnesses for complex generic APIs demands domain expertise. deepSURF automates harness generation using LLMs: it first applies static analysis to identify `unsafe` code reachability and surface the relevant API signatures, then queries an LLM to synthesise libFuzzer harnesses that instantiate the correct types (including generics via custom trait implementations). A secondary LLM pass dynamically augments harnesses to improve coverage of complex API interaction patterns. Evaluated on 63 real-world Rust crates, deepSURF rediscovered all 30 known memory-safety bugs in the test set and found 12 previously-unknown vulnerabilities, of which 11 received RustSec advisory IDs and 3 have already been patched upstream.

**Methodology:** Static analysis for `unsafe` reachability → LLM harness synthesis (handling generics via trait substitution) → LLM-augmented dynamic coverage improvement → libFuzzer execution; evaluated on 63 Rust crates.

**Why it matters for your work:** Demonstrates the LLM-augmented fuzzing synergy in a memory-safe language ecosystem; the harness-synthesis methodology adapts directly to other typed languages where API surface is hard to cover manually.

### 5. [MPC-Patch-Bench: Security-Aware LLM Code Patch for Multi-Party Computation](https://arxiv.org/abs/2606.11416)
**Authors:** Yukuan Zhang, Mengxin Zheng, Qian Lou · University of Central Florida · arXiv cs.CR/cs.SE · submitted 2026-06-09

Standard code-repair benchmarks (SWE-bench, BugsInPy) measure functional correctness via test-pass rates. For cryptographic software this is insufficient: a patch can pass every unit test while violating a security invariant — for example, introducing key reuse in an MPC protocol or accumulating floating-point error that biases a secret-share reconstruction. MPC-Patch-Bench introduces a repository-level benchmark for Secure Multi-Party Computation libraries built around a domain-specific MPC Verifier that checks both functional and cryptographic/numerical-fidelity correctness. On 100 curated repair tasks drawn from real MPC codebases, the strongest evaluated LLM achieves 22.9% functional resolution — and the MPC Verifier rejects up to 40% of those, reducing the verified resolution rate to 17.1%. The gap quantifies exactly how much current models miss when the correctness criterion includes security properties.

**Methodology:** MPC-Patch-Bench curation pipeline (domain-specific + human-AI completion); MPC Verifier combining cryptographic property checks and numerical-fidelity assertions; evaluated on 12 frontier LLMs.

**Why it matters for your work:** Establishes a template for extending code-repair evaluation to domains where functional tests are necessary but not sufficient for security — directly applicable to any security-critical software repair research.

## Also notable

### [SABER: Benchmarking Operational Safety of LLM Coding Agents in Stateful Project Workspaces](https://arxiv.org/abs/2606.01317)
**Authors:** Qi Hu, Yifeng Tang, Qinghua Wang, Lanyang Zhao, Pengji Zhang, Yuhao Qing, Xin Yao, Dong Huang, Lin Zhang, Zhuoran Ji · University of Hong Kong, Shandong University, CMU, NUS, HKUST · arXiv cs.SE/cs.CR · submitted 2026-05-31 *(missed from prior digest)*

Unlike prompt-refusal benchmarks that record what a model says, SABER evaluates what an LLM-driven coding agent actually does to the filesystem, git history, and configuration files after a sequence of actions in a Docker-sandboxed, realistic project environment. Violation categories include credential exposure, dependency tampering, build-artifact corruption, and git-history rewriting. Even the best-performing model has a harmful safety-violation rate (HSR) above 54%, revealing that instruction-following alignment does not transfer to multi-step operational contexts. The per-category breakdown enables model-specific safety profiling beyond aggregate scores.

**Methodology:** Agent-style Docker environments (source code + config + git history); environment-state evaluation after action sequences; categorised HSR metric; multi-institutional evaluation of frontier models.

### [AI Code Sandboxes: A Comparative Security Study Part 1 — Engine-Level Properties](https://arxiv.org/abs/2606.08433)
**Authors:** George Andronchik, Pavel Lokhmakov · fellows.tech / orbitalab.dev · arXiv cs.CR · submitted 2026-06-07

Systematic security comparison of five AI sandbox products (spanning microVM, userspace-kernel, and OCI-container engine classes) across six engine-level axes: host attack surface, information leakage potential, defense-in-depth stackability, public CVE history, patch cadence, and upstream fuzzing posture. Products within the same engine class diverge substantially on every axis, meaning engine architecture alone does not predict security posture — operator-facing pin policy emerges as the dominant variable. No single axis is sufficient for a comparative judgment; the paper argues the cross-axis reading is load-bearing for deployment decisions. Part 2 (runtime behaviour under adversarial workloads) is forthcoming.

**Methodology:** Six-dimensional engine-level measurement framework; CVE history and patch-cadence analysis for each product; static attack-surface enumeration by engine class.

### [Fixing Security Vulnerabilities with Agentic AI in OSS-Fuzz](https://arxiv.org/abs/2411.03346)
**Authors:** Yuntong Zhang, Sungmin Kang, Juyeon Yoon, Abhik Roychoudhury, et al. · NUS, KAIST · ICSE 2026 SEIP (presented April 12–18, 2026)

First study of LLM-assisted security patching directly in the OSS-Fuzz continuous fuzzing infrastructure. The authors adapt AutoCodeRover — an agent normally driven by issue-description text — to the security domain by replacing issue text with vulnerability-relevant code elements extracted through concolic execution of the crashing input, augmented with static type information. On a historical benchmark of confirmed OSS-Fuzz vulnerabilities, the agent generates plausible patches for 61–72% of cases, and in a live evaluation against current unpatched OSS-Fuzz reports it performs comparably to the benchmark results. The deployment pipeline integrates directly with OSS-Fuzz's build/test infrastructure.

**Methodology:** AutoCodeRover agent adapted for security domain; concolic execution of crash input to extract relevant code elements; static type augmentation; live OSS-Fuzz integration evaluated on historical and live unpatched bugs.

### [Demystifying and Detecting Agentic Workflow Injection Vulnerabilities in GitHub Actions](https://arxiv.org/abs/2605.07135)
**Authors:** Not fully identified in indexed sources · arXiv cs.CR/cs.SE · 2026-05

Systematic study of how prompt-injection payloads reach AI agents embedded in GitHub Actions workflows: taxonomy of injection sources (PR metadata, issue bodies, commit messages, file contents), propagation paths through workflow YAML to agent context, and a static analysis tool for flagging dangerous data flows before deployment. Directly complements GitInject (Top Pick #1) by providing a detection methodology where GitInject documents the attack chains. The static analyser is publicly available and operates on workflow YAML without requiring agent execution.

**Methodology:** Taxonomy of injection source × propagation path pairs; static workflow-YAML analysis for dangerous data-flow patterns; evaluated on a corpus of public Actions workflows.
