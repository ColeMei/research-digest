## Top picks

### 1. [The sorry state of skill distribution](https://blog.trailofbits.com/2026/06/03/the-sorry-state-of-skill-distribution/)
**Authors:** Samuel Judson, Tjaden Hess · Trail of Bits Blog · 2026-06-03

Trail of Bits built four malicious skills targeting Claude/MCP-style agent marketplaces and found that every major skill-distribution scanner they tested — ClawHub's detector, Cisco's agent skill scanner, and all three scanners integrated into Vercel's skills.sh — failed to flag them. Three of the four attacks took under an hour to design; the most trivial bypass prepended 100,000 blank lines to the malicious skill so the scanner truncated the file before reaching the payload, then marked it safe. Beyond code-hiding, the research shows that skills can embed harmful instructions in natural language metadata files (e.g., a `SKILL.md` prompt) that no current scanner treats as a code surface, giving an attacker a dual vector across both code and prose. The core diagnosis is structural: the agentic skill ecosystem has shipped ahead of its security supply chain, adopting a "dependency manager" model for skills without any of the security infrastructure package managers accumulated over decades.

**Methodology:** Designed four malicious skills with escalating bypass techniques; evaluated against ClawHub, Cisco, and skills.sh scanners; analyzed the scanner architectures to explain root causes.

**Why it matters:** If you are building or auditing any agent skill marketplace or tool-distribution pipeline, every supply-chain assumption you are making about upstream vetting is likely wrong — this is a direct supply-chain threat to multi-agent security tooling.

### 2. [Description-Code Inconsistency in Real-world MCP Servers: Measurement, Detection, and Security Implications](https://arxiv.org/abs/2606.04769)
**Authors:** Yutao Shi, Xiaohan Zhang, Xiangjing Zhang, Xihua Shen, Hui Ouyang, Huming Qiu, Mi Zhang, Min Yang · arXiv cs.CR · submitted 2026-06-03

An LLM receiving a tool call description assumes that description faithfully represents what the code does — this paper shows, at scale, that assumption frequently fails. The authors define Description–Code Inconsistency (DCI) as any mismatch between a tool's advertised capabilities or security boundaries and its actual code behavior, spanning two main classes: functionality mismatches (the code does more or less than described) and undeclared side effects (network calls, credential access, file writes not mentioned in the description). Applying their **DCIChecker** framework — which combines structure-aware static analysis with a Direct-Reverse-Arbitration prompting method for cross-validation — to 19,200 description–code pairs from 2,214 real-world MCP servers, they find that **9.93% of pairs exhibit DCI**, constituting a widespread defense blind spot that enables everything from operational misfires to stealthy malicious behaviors invisible to any tool-description-based policy.

**Methodology:** Static analysis of 2,214 real-world MCP servers; DCIChecker combining structure-aware AST analysis with a LLM-based Direct-Reverse-Arbitration prompting scheme for cross-validation; taxonomy construction across DCI subtypes.

**Why it matters:** MCP is the dominant tool-integration layer for LLM agents; a ~10% inconsistency rate in the real-world corpus means any security policy that trusts tool descriptions is systematically bypassable without any adversarial action required.

### 3. [Caught in the Act(ivation): Toward Pre-Output and Multi-Turn Detection of Credential Exfiltration by LLM Agents](https://arxiv.org/abs/2606.04141)
**Authors:** Kargi Chauhan, Pratibha Revankar · arXiv cs.CR · submitted 2026-06-02

LLM agents that handle credentials routinely place those secrets in the same context window as untrusted retrieved content, creating a direct path for indirect prompt injection to induce exfiltration. This paper studies the problem and proposes three layered defenses. First, **activation probes** trained on intermediate model representations detect credential-seeking intent before any output token is emitted, including under held-out encoding transformations adversaries might use to evade surface-level filters. Second, **honeytokens** generated from format-specific character models are embedded at credential positions; detection is calibrated with split conformal prediction to bound false-positive rates. Third, multi-turn exfiltration — where an attacker spreads a secret across multiple messages to evade per-turn detection — is modeled as a cumulative information-flow problem, and a leakage budget is tracked across the conversation. Controlled experiments on open-weight models show that activation features cleanly separate benign and credential-seeking prompts across all three defense layers.

**Methodology:** Activation-level probing (pre-output), honeytoken generation with conformal prediction calibration, and multi-turn cumulative information-flow tracking; evaluated on open-weight models against held-out encoding transformations.

**Why it matters:** Any agentic pipeline that touches API keys, tokens, or passwords is exposed to this attack path; the pre-output activation defense is a practical technique deployable today without model retraining.

### 4. [FORGE: Multi-Agent Graduated Exploitation and Detection Engineering](https://arxiv.org/abs/2606.03453)
**Authors:** Farooq Shaikh · arXiv cs.CR/cs.AI · accepted at AgentCy Workshop, ARES 2026 · submitted 2026-06

FORGE proposes a multi-agent framework that couples graduated exploitation — progressively escalating attack steps calibrated by severity — with synchronized detection-rule generation, treating offensive and defensive engineering as a joint optimization problem. Rather than running an exploit agent in isolation and separately synthesizing detection logic, FORGE's agents share intermediate reasoning artifacts: each exploitation step informs the detection agent's rule-writing context, and detection coverage gaps feed back into the exploitation agent's prioritization of untested attack paths. The result is a system that simultaneously benchmarks vulnerability exposure and produces operational detection content, with risk prioritization applied across the full exploitation graph. The work was accepted at the AgentCy Workshop (ARES 2026), positioning it within the emerging field of agentic cybersecurity operations.

**Methodology:** Multi-agent architecture with shared intermediate reasoning; graduated exploit escalation coupled with real-time detection-rule synthesis; risk prioritization over vulnerability graph.

**Why it matters:** The co-generation of exploit and detection artifacts from the same agentic pass is a compelling framing for automated red-team/blue-team workflows and directly addresses the "exploit found, now what?" gap in current agentic security tools.

### 5. [Investigating and Alleviating Harm Amplification in LLM Interactions](https://arxiv.org/abs/2606.02423)
**Authors:** Ruohao Guo, Wei Xu, Alan Ritter · Georgia Institute of Technology · arXiv cs.CR/cs.LG · submitted 2026-06-01

This paper defines and benchmarks *harm amplification*: the phenomenon where multi-turn LLM interactions enable users to achieve harmful outcomes far beyond what a single turn permits, through two mechanisms — democratization (novices gaining domain expertise step-by-step) and scaling (automating harmful operations at volumes unreachable manually). The authors introduce **HarmAmp**, a multi-turn benchmark spanning 12 risk categories grounded in real-world threat scenarios, with each instance satisfying three criteria: substantive amplification (outcome impossible in a single turn), operational specificity (actionable harmful output), and multi-turn necessity. They then propose **TrajSafe**, a proactive monitoring system that models trajectory risk rather than point-in-time turn risk, intervening before the harmful end-state is reached rather than at the moment of violation.

**Methodology:** HarmAmp benchmark construction (12 categories, multi-turn necessity criterion); TrajSafe trajectory-level risk model; comparative evaluation against per-turn safety filters.

**Why it matters:** Standard per-turn safety evaluations systematically undercount real-world risk in agentic deployments; HarmAmp is a ready-made benchmark for evaluating the multi-turn safety of any security-assistant or code-generation agent.

### [A Taxonomy of Runtime Faults in Model Context Protocol Servers](https://arxiv.org/abs/2606.05339)
**Authors:** Joshua Owotogbe, Indika Kumara, Willem-Jan van den Heuvel, Damian Andrew Tamburri, Antonio Ken Iannillo, Roberto Natella · arXiv cs.SE/cs.AI · submitted 2026-06-03

Manual analysis of 837 runtime fault threads across 473 actively maintained MCP server repositories; bottom-up open coding yielded a taxonomy of 11 top-level categories, 27 subcategories, and 73 leaf fault types. Coverage spans protocol interaction failures, tool invocation errors, schema enforcement gaps, state management failures, model-provider integration faults, and security validation breakdowns. Complements the DCI paper above (Top Pick #2) by characterising what breaks at runtime rather than at description level. Useful for building test suites or fault injection harnesses for MCP server security research.

### [SEC-bench Pro: Can Language Models Solve Long-Horizon Software Security Tasks?](https://arxiv.org/abs/2605.26548)
*(submitted 2026-05-26 — outside strict window; missed from 2026-05-31 digest, included once for completeness)*

**Authors:** Not publicly identified in indexed sources · arXiv cs.CR/cs.SE · 2026-05-26

183 validated vulnerability instances from V8 and SpiderMonkey spanning memory-safety, sandbox, JIT, and race-condition bugs, with a V8 subset representing over $1.5M in cumulative Google VRP awards. Frontier coding agents top out at **32.0% on V8 and 38.8% on SpiderMonkey** individually; the complementary two-agent union of ClaudeCode and Codex reaches 37.9%/48.8%. The benchmark targets the hardest bug-hunting surface in practice (production JS engine internals) and explicitly rejects fuzzing-harness shortcuts, requiring agents to reproduce vulnerabilities under browser-grade execution conditions. Strongest current contender on open weights is Kimi-K2.6 at 11.7% on V8.

**Methodology:** Three-phase pipeline — vulnerability collection from public CVE/VRP records, environment reconstruction via containerized browser builds, oracle-based validation against concrete PoC inputs.

**Why it matters:** The gap between the two-agent union (~40-49%) and human expert performance sets a concrete ceiling for current agentic bug hunting on well-hardened code; useful calibration for any work claiming broad-scope automated vulnerability discovery.

### [Hidden Thoughts Are Not Secret: Reasoning Trace Exposure in LLMs](https://arxiv.org/abs/2606.00642)
*(submitted 2026-05-30 — one day outside strict window; missed from 2026-05-31 digest, included once)*

**Authors:** Yu-An Lu, Ci-Yang Tsai, Yu-Lin Tsai, Raluca Ada Popa, Chia-Mu Yu · National Yang Ming Chiao Tung University; UC Berkeley · arXiv cs.CR · 2026-05-30

Many deployed reasoning systems suppress internal chain-of-thought traces from users, exposing only summaries. The authors introduce **REP** (Reasoning Exposure Prompting), a lightweight few-shot prompting technique that elicits *exposed* reasoning traces from these systems — traces that contain useful supervision signal despite the interface-level suppression. REP-conditioned exposed traces successfully improve smaller student models through distillation, even when the victim's internal reasoning is never directly accessible. The attack works across multiple prompting formats, demonstration sources, and victim model families.

**Methodology:** REP prompting (few-shot elicitation of exposed traces); distillation experiments comparing benign, REP-conditioned, and raw internal traces across model families.

**Why it matters:** Security systems or auditing tools that rely on chain-of-thought traces being hidden at the API boundary cannot assume that suppression equals confidentiality; the interface control is breakable by an ordinary user with no special access.
