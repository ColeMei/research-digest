## Top Picks

### 1. Code-Augur: Agentic Vulnerability Detection via Specification Inference
**[arXiv:2606.18619](https://arxiv.org/abs/2606.18619) · Zhengxiong Luo, Mehtab Zafar, Dylan Wolff, Abhik Roychoudhury · June 17, 2026 · cs.CR / cs.AI / cs.SE**

The central argument is that today's agentic auditors fail silently: the LLM concludes "no bug here" without committing the invariants that underlie that conclusion, so when those invariants are wrong nobody knows. Code-Augur enforces a *security-specification-first* discipline — when the agent deems a component secure, it must write its tacit assumptions as explicit in-source assertions. A coverage-guided fuzzer then runs concurrently to falsify those assertions; a triggered assertion either exposes a real vulnerability or a flawed specification that the agent must refine. The system found 22 previously unknown vulnerabilities in major open-source projects during evaluation.

**Methodology:** LLM agent infers security invariants per code component and writes them as runtime assertions; coverage-guided fuzzer attempts falsification; iterative refinement loop until convergence or confirmed PoC.

**Why it matters for your work:** The spec-infer → fuzz loop is among the most principled approaches yet to the false-negative problem in AI-driven auditing — directly informs how to design trustworthy, auditable agentic security tools.

### 2. OpenAnt: LLM-Powered Vulnerability Discovery Through Code Decomposition, Adversarial Verification, and Dynamic Testing
**[arXiv:2606.19149](https://arxiv.org/abs/2606.19149) · Nahum Korda, Gadi Evron (Knostic) · June 18, 2026 · cs.CR**

OpenAnt is a fully open-sourced end-to-end LLM vulnerability discovery system designed to operate at repository scale, where naïve LLM application fails due to context-window limits, cost, and unacceptably high false-positive rates. The architecture chains three stages: (1) decomposition — the repo is split into call-graph-aligned chunks sized to the model's reasoning window; (2) LLM-based static reasoning — Claude Opus 4.6 identifies candidate vulnerabilities with an explicit low false-positive budget; (3) adversarial dynamic verification — candidates are subjected to targeted test-case generation to produce crash-triggering inputs before anything is reported. The paper is accompanied by the full open-source release on GitHub.

**Methodology:** Repository chunking + call-graph construction → LLM static reasoning (Claude Opus 4.6) → adversarial verification → dynamic testing / PoC generation; open-source at [github.com/knostic/OpenAnt](https://github.com/knostic/OpenAnt).

**Why it matters for your work:** One of the first fully open-sourced repo-scale LLM vulnerability discovery pipelines — directly reproducible and extendable as a baseline for your research.

### 3. Rapid Poison: Practical Poisoning Attacks Against the Rapid Response Framework
**[arXiv:2606.16242](https://arxiv.org/abs/2606.16242) · David Huang, Jaewon Chang, Avidan Shah, Prateek Mittal, Chawin Sitawarin · June 15, 2026 · cs.CR**

The Rapid Response (RR) framework is a production system that continuously retrains jailbreak-detection classifiers from newly observed jailbreak samples — it underpins Anthropic's ASL-3 safeguards. This paper demonstrates that an adversary who can inject a single crafted message into the RR ingestion pipeline can poison the classifier's training data. Two attack objectives are mounted: (1) *targeted false-positive* attacks that cause the classifier to block innocuous inputs sharing a specific surface feature, and (2) *concept-based backdoor* attacks that induce false negatives generalizing to jailbreak families the defender has not yet seen. Both succeed at a 1% poisoning rate, pushing FPR to 100% and FNR to 96%.

**Methodology:** Adversary crafts prompt-injection payloads delivered via the RR ingestion pipeline; evaluates targeted and concept-based backdoor poisoning on a retrained binary jailbreak classifier; tests generalization to held-out jailbreak strategies.

**Why it matters for your work:** Shows that AI safety pipelines are themselves attackable attack surfaces — essential reading for anyone building or evaluating automated AI security tooling.

### 4. Calibration Without Comprehension: Diagnosing the Limits of Fine-Tuning LLMs for Vulnerability Detection in Systems Software
**[arXiv:2606.20502](https://arxiv.org/abs/2606.20502) · Arastoo Zibaeirad, Marco Vieira · June 18, 2026 · cs.CR**

This paper questions whether LLMs that score well on vulnerability benchmarks are actually reasoning about security, or merely pattern-matching on benchmark contamination. The authors build CWE-Trace — 834 manually curated Linux kernel samples spanning 74 CWEs — with a strict temporal train/test split to eliminate leakage from public CVE disclosures. Under these conditions the best fine-tuned model reaches only 52.1% detection accuracy; calibration analysis (ECE and reliability diagrams) reveals overconfident, poorly calibrated predictions, suggesting models correlate surface-level code features to labels rather than understanding vulnerability semantics.

**Methodology:** CWE-Trace benchmark construction with temporal split; fine-tuning and zero-shot evaluation of multiple LLMs; calibration analysis alongside detection metrics (precision, recall, F1, AUC).

**Why it matters for your work:** If you evaluate or build LLM-based vulnerability detection tools, this is a necessary methodological caution — temporal splits and calibration analysis should be table-stakes for any honest evaluation.

### 5. PUFFERDOS: Efficient and Effective Attack String Generation for Regular Expression Denial of Service Vulnerabilities
**[arXiv:2606.19654](https://arxiv.org/abs/2606.19654) · Shangzhi Xu, Ziqi Ding, Xiao Cheng, Yuekang Li, Nan Sun, Benjamin Turnbull, Shuangxiang Kan, Siqi Ma · IEEE S&P 2026 · June 17, 2026 · cs.CR**

ReDoS exploits the super-polynomial backtracking behavior of regex engines to deny service with short malicious inputs. Prior tools like RENGAR search for attack strings heuristically; PUFFERDOS instead starts from formal characterization of the regex structural patterns that cause worst-case growth, then uses ReDoS-specific compositional concolic execution to guarantee that generated strings actually trigger the vulnerability. Compared to RENGAR, PUFFERDOS achieves 97.2×–3,872.4× efficiency improvements on existing vulnerable regexes and discovered 59 exploitable ReDoS instances across 12 popular open-source projects (25 more than RENGAR).

**Methodology:** Formal regex vulnerability classification → compositional concolic execution for attack string synthesis → empirical evaluation on real-world regex datasets and 12 popular OSS codebases.

**Why it matters for your work:** Accepted at S&P'26; a strong tool for automated vulnerability discovery in software pipelines, and a solid formal-methods baseline for the program-analysis side of AI-empowered security.

## Also Notable

### NRT-Bench: Benchmarking Multi-Turn Red-Teaming of LLM Operator Agents in Safety-Critical Control Rooms
**[arXiv:2606.20408](https://arxiv.org/abs/2606.20408) · Hanwool Lee et al. · June 18, 2026 · cs.CR**

NRT-Bench instantiates a five-role LLM operator team (each role backed by a configurable frontier model) to run a simulated nuclear power plant governed by six critical safety functions (CSFs). Adversaries inject messages over four communication channels in bounded multi-turn sessions with per-turn feedback. Crucially, harm is an *objective signal* — a run terminates and is scored the moment any CSF is lost — rather than relying on LLM-judged text quality. Evaluating four frontier operator models, 8.7–12.1% of attack sessions result in CSF loss; counterintuitively, the same guardrail stack that reduces attack success for one model raises it for another.

**Methodology:** Multi-agent simulation (5 LLM operator roles + adversarial attacker); objective CSF-loss termination signal; four frontier model configurations; ablation of guardrail stacks and safety-advisor agent.

**Why it matters for your work:** The objective harm signal design is a methodological contribution worth adopting in your own multi-agent security evaluations.

### A Red-Team Study of Anthropic Fable 5 & Opus 4.8 Models
**[arXiv:2606.18193](https://arxiv.org/abs/2606.18193) · June 16, 2026 · cs.CR**

An independent adversarial evaluation of Anthropic's two latest frontier models across 7,826 harmful intents spanning ten harm categories, using the HackAgent red-teaming framework to mount four families of automated jailbreak attacks (hundreds of thousands of adversarial attempts total). Both models resist the majority of attacks; the residual attack surface is dominated by *adaptive iterative* attacks while static obfuscation is near-fully neutralized. All apparent successes were re-adjudicated by a three-model judge panel (majority vote) to control for false positives from weak judges.

**Methodology:** HackAgent toolkit; four automated jailbreak families (static obfuscation, adaptive iterative, multi-turn, composite); 7,826 harmful intents; three-model judge panel re-adjudication.

**Why it matters for your work:** Up-to-date picture of frontier LLM robustness — useful for calibrating red-team effort, setting benchmark difficulty, and understanding which attack families still warrant attention.
