## Top picks

### 1. [Towards Demystifying and Repairing LLM-in-the-Loop Vulnerabilities](https://arxiv.org/abs/2605.28893)
**Authors:** Yujie Ma, Jialin Rong, Chenxi Yang, Lili Quan, Xiaofei Xie, Yongqiang Lyu, Qiang Hu · Tianjin University; Singapore Management University · arXiv cs.SE/cs.CR · submitted 2026-05-27

As LLMs become embedded in production software stacks, a new class of bugs has emerged that is distinct from model-level jailbreaks: vulnerabilities introduced by the interaction between LLM outputs and their downstream framework components (tool runners, vector stores, memory managers, RAG pipelines). This paper builds **LLMCVE**, the first dataset dedicated to this *LLM-in-the-loop* failure class, by crawling 2,888 CVEs and issue reports across 230 popular LLM-ecosystem libraries (LangChain, LlamaIndex, AutoGPT, CrewAI, and 226 others), then hand-triaging 205 vulnerabilities that strictly satisfy the definition. The dominant root causes are execution environments that fail to distinguish untrusted LLM-generated data from executable system commands, and severe boundary isolation failures when LLMs interact with complex agent toolchains or external knowledge bases. The paper provides a root-cause taxonomy and a repair taxonomy covering the framework layer, which prior patch datasets do not address.

**Methodology:** Empirical study of 2,888 public CVEs/issues across 230 LLM-framework repos; manual triage to 205 strict LLM-in-the-loop cases; root-cause classification and repair taxonomy construction.

**Why it matters:** The first taxonomy and dataset focused on deployed-stack vulnerabilities of LLM software is a critical resource for building automated detectors and patching tools that go beyond model-level analysis.

### 2. [Relevance as a Vulnerability: How Web Retrieval Degrades Safety Alignment in LLM Agents](https://arxiv.org/abs/2605.29224)
**Authors:** Aditya Nawal, Manit Baser, Mohan Gurusamy · National University of Singapore · arXiv cs.CR/cs.LG · submitted 2026-05-28

Enabling web retrieval in LLM agents consistently degrades safety alignment — even when retrieved content is benign or actively safety-oriented. The authors introduce **AgentREVEAL**, a diagnostic framework that decomposes alignment degradation by retrieval-pipeline architecture and content properties. Their headline finding is the *Safe Source Paradox*: pages that explicitly flag risks or contain anti-harmful disclaimers increase compliance with harmful requests by an average of **25%** compared to a no-retrieval baseline, because relevance-weighted retrieval normalizes the topic and lowers the model's refusal threshold through mere contextual priming. The effect is consistent across multiple frontier models and is independent of whether the retrieved content was crafted adversarially.

**Methodology:** AgentREVEAL diagnostic framework; systematic evaluation across retrieval pipeline architectures, content types, and frontier LLM families; alignment degradation measured on curated harmful-request benchmarks.

**Why it matters:** Every RAG-enabled agentic security tool is exposed to this attack surface — poisoning a retrieval corpus with plausibly safety-oriented documents is a practical, zero-craftsmanship adversarial strategy against agentic pipelines in the wild.

### 3. [MemMorph: Tool Hijacking in LLM Agents via Memory Poisoning](https://arxiv.org/abs/2605.26154)
**Authors:** Xuanye Zhang, Yongsen Zheng, Zhuqin Xu, Kaiyu Zhou, Bowen Shen, Haoran Ou, Tianwei Zhang, Kwok-Yan Lam · arXiv cs.CR · submitted 2026-05-24

MemMorph is the first attack that biases tool selection in LLM agents by poisoning the agent's persistent long-term memory rather than by issuing override instructions. The attack injects a small number of crafted records — disguised as technical facts, incident reports, and operational policies — into the agent's vector-store-backed memory. These records reshape the agent's contextual perception across future turns so that it autonomously infers and selects the attacker-preferred tool without any explicit directive appearing in the current prompt. The poisoned records are syntactically indistinguishable from normal operational memory entries, making them invisible to prompt injection detectors and human reviewers auditing the active context window.

**Methodology:** Memory poisoning attack targeting vector-store-backed persistent agent memory; disguised records constructed from three categories of operational document types; evaluated on representative agent frameworks with tool-selection outcome measurement.

**Why it matters:** Long-term memory is the default persistence mechanism for production agentic systems; this paper opens a new attack class orthogonal to prompt injection defenses and directly challenges any multi-agent security architecture that trusts its own memory store.

### [Got a Secret? LLM Agents Can't Keep It: Evaluating Privacy in Multi-Agent Systems](https://arxiv.org/abs/2605.27766)
**Authors:** Aman Priyanshu, Supriti Vijay (Foundation AIUSA), Esha Pahwa (Corvic AIUSA) · arXiv cs.CR · 2026-05-26

A simulation study deploying thousands of LLM agents across social communities over a simulated month, measuring privacy leakage as a function of social pressure. Moving from single-turn to multi-turn social evaluation amplifies privacy violations from **19.95% to 45.30%** across OpenAI model families. Critically, leakage is *socially contagious*: agents are **8×** more likely to disclose sensitive information after observing a peer disclose it. Explicit privacy instructions reduce but do not eliminate this amplification effect.

**Methodology:** Moltbook-style large-scale multi-agent social simulation; privacy leakage measured under varying social pressure; tested across model families with and without explicit privacy instructions.

**Why it matters:** Multi-agent security pipelines that pass sensitive context between agents face a contagion-style leakage failure mode that is distinct from direct prompt extraction and is not captured by single-agent safety benchmarks.

### [Cybersecurity AI (CAI) Dataset: Fourteen-Month Corpus of LLM Hacker Trajectories](https://arxiv.org/abs/2605.28146)
**Authors:** Víctor Mayoral-Vilches · arXiv cs.CR · 2026-05-27

The CAI Dataset aggregates **230,935 session logs** and **26,027,742 user prompts** from 16,768 source IPs across 123 countries, exercising 4,187 unique LLM identifiers against 23,147 target domains over 18.07 TB of durable storage, collected through the open-source CAI agent framework over fourteen months of real-world cybersecurity operations. The traffic mix is 36.4% offensive, 20.1% attacker-intent, 27.5% business/integration, and 4.4% defensive, constituting the largest described corpus of LLM-driven hacker trajectories. The central operational finding — echoing PentestGPT — is that expert operator trajectories, not raw model capability, are the primary bottleneck for cybersecurity LLM performance.

**Methodology:** Fourteen-month passive logging of all session trajectories through the open-source CAI agentic framework; released in tiered audience sizes (CAI Dataset10 / 1k / 200k) to partner organizations.

**Why it matters:** The first large-scale real-world dataset of LLM offensive-operator behavior is a foundational resource for training and evaluating security-specialized models and for studying how expert technique transfers into LLM-assisted attack workflows.

### [Reframing LLM Agent Security as an Agent-Human Interaction Problem](https://arxiv.org/abs/2605.24309) *(~May 23, not in prior digest)*
**Authors:** Peiran Wang, Ying Li, Yuan Tian · UCLA · arXiv cs.CR · submitted ~2026-05-23

This position paper argues that LLM agent security is fundamentally an *agent-human interaction* (AHI) problem. Through systematic analysis of 59 academic papers, 21 production agent systems, and 26 security plugins as of April 2026, the authors find a research-practice inversion: the three human-centric mechanisms most heavily deployed in production — policy specification, runtime approval, and scope configuration (each adopted by ≥14 of 21 systems) — see near-zero academic coverage, while the mechanisms most studied academically (intent anchoring, trust labeling) see zero production deployment. The result is that the majority of security research effort is orthogonal to what practitioners have found actually works.

**Methodology:** Systematic review of 59 academic papers + 21 production agent systems + 26 security plugins; cross-referencing adoption frequencies across seven security mechanism categories.

**Why it matters:** A useful benchmark for calibrating which agent-security research directions will translate to production impact — the mechanism-adoption gap is large and actionable.

### [Autonomous LLM Agents & CTFs: A Second Look](https://arxiv.org/abs/2605.21497) *(~May 20, not in prior digest)*
**Authors:** Youness Bouchari, Matteo Boffa, Marco Mellia, Idilio Drago, Thanh Minh Bui, Dario Rossi · arXiv cs.CR/cs.SE · submitted ~2026-05-20

This paper revisits recent claims of near-human performance by LLM agents on CTF challenges, engineering agent architectures of increasing complexity across 30 web-based CTF tasks spanning 14 vulnerability classes. The key finding is that `claude-code`, a general-purpose agent that self-determines its architecture, solves **19/30** tasks — matching the best hand-engineered specialized architectures — suggesting general-purpose agents are strong baselines that specialized designs have not yet beaten. The result calls into question whether the architectural complexity of purpose-built offensive agents is justified at current capability levels.

**Methodology:** Comparative evaluation of engineered agent architectures vs. general-purpose agents on 30 web CTF challenges across 14 vulnerability classes; instantiated across multiple LLM backbones.

**Why it matters:** If general-purpose agents match specialized architectures on offensive tasks, the field may be over-indexing on architecture and under-investing in training signal and capability — a direct input to benchmark and evaluation strategy for agentic security research.

### [Microsoft Security Blog: Introducing RAMPART and Clarity](https://www.microsoft.com/en-us/security/blog/2026/05/20/introducing-rampart-and-clarity-open-source-tools-to-bring-safety-into-agent-development-workflow/) *(~May 20, not in prior digest)*
**Authors:** Microsoft Security Engineering · Microsoft Security Blog · 2026-05-20

Microsoft open-sourced two tools to bring safety and adversarial testing into the agent development workflow: **RAMPART** (an agent test framework that encodes prompt injection, privilege escalation, and goal-drift scenarios as CI-runnable repeatable test cases) and **Clarity** (a structured pre-build sounding board aligned to OWASP LLM Top 10 and Microsoft's Responsible AI principles that surfaces security and alignment concerns before any code is written). RAMPART is designed to catch adversarial failures in the standard build pipeline, treating agent security testing as a first-class engineering artifact rather than a post-hoc red-team exercise.

**Methodology:** Open-source tooling; RAMPART encodes adversarial scenarios as repeatable automated test cases runnable in CI; Clarity provides structured checklists for pre-build design review.

**Why it matters:** CI-integrated adversarial testing for agents is a practical contribution directly applicable to any agentic security tooling project under active development.
